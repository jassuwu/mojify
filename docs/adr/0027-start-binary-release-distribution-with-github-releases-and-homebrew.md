# Start binary release distribution with GitHub Releases and Homebrew

Mojify will make its first installable CLI distribution stage binary-first rather than npm-first: stable SemVer-compatible calendar build tags in the shape `v0.YYYYMMDD.BUILD` will publish prebuilt macOS and Linux CLI tarballs plus checksums through GitHub Releases, and GoReleaser will update a separate public `jassuwu/homebrew-tap` formula for `brew install jassuwu/tap/mojify`. GoReleaser's formula publishing is deprecated in favor of casks, but Mojify still uses it in this stage because the desired Homebrew artifact is a CLI formula rather than a desktop app cask. The first binary matrix is `darwin/amd64`, `darwin/arm64`, `linux/amd64`, and `linux/arm64`. Real publishing is tag-only on GitHub-hosted runners; manual workflow dispatch and scheduled publishing are deferred, while local snapshot QA covers token-free release dry runs without publishing or touching the Homebrew tap. Main CI should run a non-publishing GoReleaser snapshot check so broken release packaging is caught before a tag, and the tag workflow should still rerun the normal Go tests plus a version-output smoke check before publishing. `mojify --version` and `mojify version` should both print a single user-facing line like `mojify 0.20260602.145`; commit and build date may be injected for internal release tooling, but verbose diagnostics are deferred. GitHub Releases should use generated release notes for this stage rather than a hand-maintained changelog. GoReleaser is scoped to CLI binary distribution for this stage; future npm wrappers, desktop app packaging, and landing-site deployment can use separate release tooling when those surfaces exist. Creating the tap is a manual operator setup step, not an automated side effect of the Mojify release workflow, and the workflow should assume the tap already exists rather than creating external repositories. The release runbook should include step-by-step tap creation, release setup, and smoke-test instructions, but remain public-safe: it may document repository names, secret names, token purpose, and least-privilege scope, but never token values or private account material. Windows support is WSL-only for this stage, Linux distro packages, native Windows binaries, `go install` as an advertised install path, signing, notarization, shell completions, manpages, and cosign-style provenance are deferred; released artifacts should publish checksums only. Homebrew tap publishing uses a dedicated `TAP_GITHUB_TOKEN` scoped to the tap repo rather than relying on the main repository `GITHUB_TOKEN`, Homebrew installs should declare `ffmpeg` and `yt-dlp` as formula dependencies and install only the `mojify` binary, tarball installs should document those same runtime tools as prerequisites rather than bundling or installing them at Mojify runtime, and missing-tool handling should stay as command-specific runtime dependency hints rather than a new `doctor` command.
