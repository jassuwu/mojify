# Mojify README Header Demo Design

## Status

Approved concept: **CLI Tunnel**.

This spec captures the design for Mojify's README header demo asset. It does not implement the asset pipeline yet.

## Goal

Create a short animated README header that proves Mojify's core promise in one glance:

> Turn media into text.

The header should feel like a terminal-first product without becoming a busy terminal UI screenshot. It should show a polished source animation with large `mojify` text, a CLI transformation beat, and a final Mojify-rendered output reveal.

## Non-Goals

- Do not use copyrighted or ambiguous sample footage.
- Do not make a broad feature montage of every Mojify command.
- Do not make a pure logo loop that fails to show Mojify's product behavior.
- Do not put the asset pipeline in default CI yet.
- Do not claim the final composite GIF is pure Mojify output if FFmpeg also composes the source, command beat, and output reveal.

## Visual Story

The final GIF should follow a simple three-beat loop:

1. **Source beat**
   - A Remotion-generated source animation appears on a dark, self-contained background.
   - Large `mojify` text is the dominant visual.
   - Motion is clean and readable: slide, scale, scanline, light sweep, or subtle perspective movement.

2. **CLI tunnel beat**
   - The source animation visually passes through a CLI command layer.
   - The command should be legible but not tiny:

   ```text
   $ mojify export source.mp4 header.gif
   ```

   - The command beat should communicate transformation without becoming a full terminal mockup.

3. **Mojify output reveal**
   - The final frame becomes colored character-video output generated by Mojify.
   - The word `mojify` remains recognizable in the rendered output.
   - The loop should land on a readable final state before restarting.

The CLI tunnel is the primary story. A brief source/output overlap or comparison can appear during the transition, but the header should not end as a permanent 50/50 comparison.

## Asset Shape

Commit these final assets:

- `docs/assets/readme/mojify-header.gif`
- `docs/assets/readme/mojify-header-poster.png`

Embed near the top of `README.md`:

```html
<p align="center">
  <picture>
    <source media="(prefers-reduced-motion: reduce)" srcset="docs/assets/readme/mojify-header-poster.png">
    <img alt="Mojify transforms a polished mojify source animation into colored text video output." src="docs/assets/readme/mojify-header.gif" width="960">
  </picture>
</p>
```

The poster should be a clear still from the final Mojify output reveal.

## Pipeline

The asset generation pipeline should be manual and reproducible:

```text
Remotion source animation
  -> dist/readme-header/source.mp4
  -> ./bin/mojify export ... dist/readme-header/mojify-output.mp4
  -> FFmpeg composition and palette optimization
  -> docs/assets/readme/mojify-header.gif
  -> docs/assets/readme/mojify-header-poster.png
```

Recommended command shape for the Mojify proof output:

```bash
./bin/mojify export \
  --overwrite \
  --width 480 \
  --fps 12 \
  --duration 4s \
  dist/readme-header/source.mp4 \
  dist/readme-header/mojify-output.mp4
```

The implementation can tune width, FPS, and duration if actual readability or file size demands it.

## Repository Layout

Use the monorepo shape already reserved by the project:

```text
apps/readme-header/              # Remotion source animation app
scripts/render-readme-header.sh  # manual render pipeline
dist/readme-header/              # ignored intermediates
docs/assets/readme/              # committed README assets
```

Add a root script for manual use. Preferred name:

```json
"readme:header": "bash scripts/render-readme-header.sh"
```

The script should build or locate `./bin/mojify`, render the source animation, export Mojify output, compose the final GIF, create the poster, and print file sizes.

## Remotion Scope

Use Remotion only for the original source animation. Mojify owns the transformation proof. FFmpeg owns final composition, timing, and GIF optimization.

If Remotion is added, scaffold it with Remotion's official tooling rather than hand-writing a full app structure. Keep it isolated under `apps/readme-header/`.

Remotion should stay out of the default build/test/CI path for this stage. The README header render is an opt-in asset command.

Before committing the dependency, verify Remotion's current license terms and document any relevant non-MIT implications in the implementation plan or PR.

## Asset Constraints

- Width: `960px` final GIF.
- Height: `240-360px`.
- Duration: `3-4s`.
- FPS: `10-12fps`; use `15fps` only if file size stays low.
- File size target: `<= 2MB`.
- File size hard tolerance: `<= 5MB`.
- Background: dark and self-contained so it works in GitHub light and dark themes.
- Text: large enough to remain readable after Mojify rendering and GIF compression.
- Motion: simple, with no flashing or noisy high-frequency patterns.

## QA

Manual QA should check:

- `mojify` is readable in the source beat.
- The CLI command is readable without requiring zoom.
- The Mojify output reveal is visibly character-rendered.
- The loop does not flash harshly.
- The final GIF stays under the size tolerance.
- The README renders the GIF and reduced-motion poster path correctly.
- The generated source and intermediate files remain ignored under `dist/readme-header/`.

Useful commands:

```bash
bun run build
bun run readme:header
ls -lh docs/assets/readme/mojify-header.gif docs/assets/readme/mojify-header-poster.png
```

## Risks

- GIF size can balloon if the source animation uses gradients, noise, or fast movement.
- Small terminal chrome will disappear after Mojify rendering and GIF compression.
- The composite header will be a proof asset that includes Mojify output, not a pure direct Mojify export.
- GitHub image caching may lag after replacing a published GIF.
- Binary release tarballs may include `README.md` without `docs/assets/readme/` unless release packaging later accounts for README-linked assets.

## Acceptance Criteria

- README contains the animated header near the top.
- Final committed assets are original and safe to distribute.
- Final GIF uses Mojify-generated output as the reveal.
- Final GIF target constraints are met or any deviation is explicitly documented.
- The render pipeline can be rerun locally with one manual command.
- Normal project build/test commands do not require Remotion rendering.
